## 2. 项目简介

本项目实现一个中国象棋 AI，用于在给定棋局状态下进行博弈决策。核心目标是通过搜索算法与多维度局面评估函数，结合图形化界面，实现人机对弈和算法效果验证。

### 2.1 作业目标

- 实现坚实的基础对弈逻辑框架（棋盘、棋子、合法步生成、胜负判定）
- 设计并实现多维度的复合局面评估函数（静态+动态）
- 构建带有 Alpha-Beta 剪枝的 MinMax 搜索算法，并结合时间限制进行控制
- 完成直观的图形化 UI 界面，实现完整的人机对弈交互流程

### 2.2 功能概述

- 棋局状态表示与落子规则实现
- 合法走法生成验证系统
- Minimax Min-Max 搜索与 Alpha-Beta 剪枝
- 基于超时截断的搜索时间限制策略
- Tkinter 桌面端图形可视化对弈界面

## 6. 核心算法设计

### 6.1 状态表示

- **棋盘编码方式**：使用 10 行 $\times$ 9 列的二维列表（List of Lists）表示棋盘。
- **棋子类型表示**：使用 2 字符字符串标记，第一位代表阵营（`r`为红，`b`为黑），第二位代表子力（如 `K`帅、`A`仕、`C`炮 等）。无棋子的地方标记为 `.`。
- **当前行棋方表示**：定义全局常量 `RED = 1` 和 `BLACK = -1`。

### 6.2 走法生成

- **各棋子走法规则实现方法**：在 `basic/moves.py` 中，根据棋子类型采用不同的位移增量列表。专门处理了"马走日（蹩马腿）"和"炮隔子打"的视线阻挡判定。
- **将军与胜负规则**：在 `basic/rules.py` 中检查双方将帅是否处于对方合法走法的死角。并在每次移动后检查是否"游戏结束（Winner判定）"。

### 6.3 搜索算法

- **使用算法**：Minimax 结合 Alpha-Beta 剪枝。
- **搜索控制**：原本使用绝对搜索深度。为保证 UI 响应流畅加入了 `time_limit`（默认3秒）的时间限制。递归时动态核验耗时，触发超时立刻阻断，用现有评估返回结果或兜底用启发式选首步。
- **着法排序（Move Ordering）**：采用了如 `历史启发(History Heuristic)` 和吃子优先判定对生成的合法步提前排序（`_sort_moves_with_heuristics`），极大提高了 Alpha-Beta 剪枝的截断效率。

### 6.4 评估函数

可从以下维度描述（实现于 `minmax_search.py`的 `evaluate_board` 和 `minmax_ulti.py` 的细节里）：

- **动态子力价值**：基础价值随局面的变化存在动态计算方案。
- **位置价值**：对兵（过河积分补偿）和马（中心控制位积分）有着详细的加分项。
- **机动性与安全性**：引入合法步数差评分、士象防御结构完整度评估、将帅安全评估模型。
- **兵力保护关联**：在启发机制里纳入了棋子相互间的根部保护分数。

评估函数采用多维度的特征值加权累加机制，最后取总分（以红方为第一视角进行加减）。

## 7. 实验设计与结果

### 7.1 实验设置

- **对手类型**：由于本系统主打人机交互开发，实验对手主要是 人类玩家 以及 RandomAI (随机落子)。
- **超时限制**：设定在 3.0~5.0 秒。主要受制于原生 Python 以及单线程 UI 阻塞问题。

### 7.2 结果分析

- **深度提升对性能和耗时的影响**：搜索深度从2增加到3时，耗时呈指数级上升。加入时间截断控制可以完美解决这一问题，但残局中受剪枝影响搜算落点极多。
- **剪枝策略的收益**：结合着法排序模块后的 Alpha-Beta 剪枝比原生纯 MinMax 至少剔除了超一半以上的无意义分支计算。

## 9. 总结与改进方向

### 9.1 本次作业收获

- 极大掌握了基础博弈算法在真实复杂二维网图游戏规则中的应用落地实践。
- 对 Python GUI 程序的开发、界面响应刷新与事件驱动流程有了具体的了解。
- 掌握了多维评估状态加权的方法及启发式排序在提高剪枝效率上的关键作用。

### 9.2 可改进方向

- **Python性能瓶颈**：可将搜索层使用 C++ 或结合 Numba、Pypy 编译提高递归速度，进而可以探寻深度 5~7 层甚至更高的性能。
- **引入迭代加深（Iterative Deepening）**：目前使用的超时截断机制遇到当层搜索了一半时只能退回，使用迭代加深能在超时情况下完美保存上一层的完整计算结果。
- **多线程处理**：通过 `threading` 的方式将 UI 渲染和引擎思考彻底解耦，防止目前的“模拟阻塞”状态。
